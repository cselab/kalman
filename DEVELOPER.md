Practical reference for the Kalman-filter discovery experiments in this repo.
Read alongside `README.md` (which has the high-level pitch and SLURM/GPU notes)
and `demo/demo.py` (a self-contained inference reproduction of Tables 1 & 2).

## State-space model

Constants are repeated verbatim in every experiment script:

```python
dim = 2
F  = [[1, 1], [0, 1]]            # linear advection
cQ = [[0.5, 0], [1, 0]]          # process-noise factor
Q  = cQ @ cQ.T
H  = R = I (2x2)                 # observation
```

Trajectory length 500. The four scenarios differ only in the trajectory
generator (and, for non-linear, in the predict step):

| Scenario | Generator difference |
|----------|---------------------|
| Gaussian (Table 1)          | `x = F x + cQ ε`, `ε ~ N(0,1)` |
| Half-normal (Table 2)       | `ε := |ε|` for both process and obs |
| Delayed observations        | `F_t = [[1, dt], [0, 1]]`, `dt = 1 + U(0.01, 0.3)` |
| Non-linear                  | `x ← x + ϕ(x) + noise`, `ϕ(x) = [0.05 x₀³ − 2 x₀, 0.1 sin x₁]` |

In the non-linear scenario the Kalman *baseline* is an EKF that applies `ϕ`
inside the predict step — `demo/demo.py` passes `phi=phi_nl` to `kalman_filter`
to match the paper's number.

## Seed convention

Test seeds are always **32..81 (50 trajectories)**. Validation seeds differ:

| Pipeline                              | Validation seeds |
|---------------------------------------|------------------|
| Table 1 FunSearch (`PureFunsearch_*`) | `100..129` (30 traj) |
| Table 2 FunSearch (`fullkalmanfilter*`) | `12..61` (50 traj) |
| CGP (both tables)                     | drawn during search; test uses 32..81 |

The demo uses test seeds (32..81) so its numbers are directly comparable to the
paper's reported Tables 1 & 2.

## CGP pipeline

Each CGP experiment is a single `train.py` (or `predict.py`) launched as a
SLURM array (`train.sh` / `script.sh`, typically `--array=1-5` or `1-15`).
Each array task is an independent restart. The driver:

1. Imports `gp.py` (small CGP core: random/mutate/build, op table).
2. Picks an op vocabulary — for Table 2 half-noise that's
   `{matmul, minus, add, transpose, inv}`.
3. Runs an island-based (1+λ)-ES; fitness = MSE against a held-out target
   (target ≈ Kalman estimate on `validation_trajectories`).
4. Writes the best genome as a graphviz `dot` string to the array's `.out`
   file. The demo parses that `dot` via `_parse_graph` + `_execute_graph`.

To reproduce a CGP result you read the winning `.out` log, copy the `dot`
string, and execute it on the test trajectories. `demo/demo.py` does exactly
this for the three Table-2 winners (`CGP_HALF`, `CGP_DELAYED`, `CGP_NONLINEAR`).

## FunSearch pipeline

Two stages, one SLURM job each.

**Stage 1 — search (`script_funsearch.sh` → `funsearch.py`).**
4× H100, runs `deepseek-ai/DeepSeek-R1-Distill-Qwen-14B` (NF4-quantized) with
one process per GPU (`--num_threads 4`). Each process samples a batch of 30
prompts from `memory.py`, asks the model to mutate them, evaluates the
returned `aproximate(...)` function on validation trajectories, and stores the
best on a heap. Logs land in `pytorch_<jobid>.out` — one line per accepted
candidate, e.g. `[Process 2] best score so far: 1.925, content: def aproximate(...)`.

**Stage 2 — validate + test (`script_evaluate_and_test.sh` → `evaluate_and_test.py`).**
Parses *all* candidates out of the chosen `pytorch_*.out`, re-scores each on
validation, picks the best by validation MSE, then reports a clean test-MSE
and Kalman baseline.

### Gotcha — printed "Best Function Code"

`evaluate_and_test.out` ends with a "Best Function Code" block. Past
experience: that printed snippet does not always reproduce the score the same
script just reported. In particular, the delayed-observations winner only
reproduces when you take the function literally from the search log
(`pytorch_18588979.out` for that experiment) — the version that uses
`np.random.poisson(0.03, x.shape)`. If a printed best looks numerically off,
grep the raw log for `best score so far:` and use the highest one.

## bipacking/

Core FunSearch tooling shared by the table experiments:

- `funsearch.py`     — LLM driver (multi-GPU, multi-process pool).
- `memory.py`        — prompt buffer with top-k retention and temperature sampling.
- `evaluate.py`      — fitness evaluator (calls the candidate on trajectories).
- `apply.py`         — applies a candidate to a single trajectory (helper).
- `generate_instances.py`, `algorithm_progression.py`, `print_instances.py` — analysis utilities.
- `script_funsearch.sh` — the canonical SBATCH wrapper.

The per-experiment dirs under `table*/Funse*/...` are *copies* with
modifications (different `evaluate.py`, different initial prompt, sometimes a
different op set). Diff against `bipacking/` to see what each scenario changes.

## Running an experiment

CGP (Table 2 half-noise, as an example):
```bash
cd table2/CGP/half_noise_full_kalman_filter
mkdir -p full_kalman_filter_output
sbatch train.sh                       # SLURM array of restarts
# winners appear in full_kalman_filter_output/fullkalmnfilte_output_*.out
```

FunSearch (Table 2 half-noise):
```bash
cd table2/Funseach/fullkalmanfilterapproximation_funseach_half_noise
sbatch script_funsearch.sh            # 3 days on 4× H100
# then, with the right pytorch_*.out filename in evaluate_and_test.py:
sbatch script_evaluate_and_test.sh
```

CPU-only smoke-test of an already-discovered filter — open `demo/demo.ipynb`
in Colab via the badge in `README.md`, or run `python demo/demo.py` locally.

## Demo / Colab notebook

`demo/demo.py` is the source of truth (jupytext "percent" format). The
notebook `demo/demo.ipynb` is generated from it and committed with executed
outputs so the Colab badge renders without re-running.

Regenerate after editing `demo.py`:

```bash
cd demo
jupytext --to ipynb demo.py
jupyter nbconvert --execute --to notebook --inplace demo.ipynb
```

Source-file references in the notebook are GitHub permalinks pinned to a
specific commit SHA — bump the SHA in `demo.py` if you want the links to
follow `main`.

What the demo does:

1. Defines the four scenarios and a vanilla Kalman baseline (with optional
   `phi` for EKF).
2. Loads three FunSearch winners (`fs_half`, `fs_delayed`, `fs_nonlinear`) and
   three CGP winners as graphviz strings.
3. **Cross-evaluates** every (method × scenario) pair — the on-diagonal cells
   are the paper's Tables 1 & 2; the off-diagonal cells (not in the paper)
   show how each discovered filter specializes.
4. Reports `mean ± stderr` over 50 test trajectories. `stderr =
   sample_std(ddof=1) / sqrt(50)`, matching `evaluate_and_test.py:184` and
   `:228` in the FunSearch eval scripts.
5. Plots a 7×4 matrix of error-magnitude traces (Kalman + 6 discovered rows ×
   4 scenarios). Diverged trajectories are skipped before plotting.

## Reproducibility status (May 2026)

Diagonal cells of the demo match the paper to three decimal places:

| Filter | Scenario | Paper | Demo |
|--------|----------|------:|-----:|
| Kalman / EKF | Gaussian / Half / Delayed / Non-lin | 0.995 / 1.622 / 10.618 / 1.023 | exact |
| FunSearch    | Half / Delayed / Non-lin             | 0.562 / 4.96 / 0.806           | 0.562 / 4.937 / 0.806 |
| CGP          | Half / Delayed / Non-lin             | 0.603 / 2.279 / 0.876          | exact |

The 4.937 vs 4.96 gap on FS-delayed is Poisson sampling noise inside the
discovered filter itself. Off-diagonal cells are new and show that each
discovered filter is tuned to its training scenario — FS non-linear diverges
on the other three.

Outstanding caveats:

- One referenced search log (`pytorch_18877820.out` in
  `table2/Funseach/.../half_noise/evaluate_and_test.py:156`) is not in the
  repo; the demo uses the function that *does* reproduce, parsed from a
  different `pytorch_*.out`.
- The CGP graphviz strings embedded in `demo.py` are extracted by hand from
  the winning `.out` files. If you re-run a CGP experiment and want to update
  the demo, copy the winner's `dot` string into the corresponding
  `CGP_*` constant.
